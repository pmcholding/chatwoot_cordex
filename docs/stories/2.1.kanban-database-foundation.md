# Story 2.1: Kanban Database Foundation and Core Models

## Status
Approved

## Story
**As a** system administrator and backend developer,  
**I want** to implement the foundational database schema and models for the Kanban conversation management system,  
**so that** conversations can be organized into customizable workflow stages with proper data integrity and performance.

## Acceptance Criteria

1. A new database table `kanban_stages` is created with proper schema, constraints, and indexes
2. The existing `conversations` table is extended with `kanban_stage_id` foreign key (nullable)
3. A `KanbanStage` ActiveRecord model is implemented with all required validations and associations
4. The `Conversation` model is extended with kanban-related functionality and associations
5. Feature flag infrastructure is added to control kanban rollout per account
6. Default kanban stages are automatically created for existing accounts during migration
7. Database performance is optimized with proper indexes for kanban queries
8. All database changes maintain backward compatibility and support safe rollback
9. Comprehensive model validations ensure data integrity and prevent invalid states
10. Real-time broadcasting infrastructure is implemented for stage changes

## Dev Notes

**Architecture Context**: This story implements Phase 1 of the Kanban System Architecture as defined in [docs/kanban-architecture.md]. It establishes the foundational data layer that all subsequent kanban features will build upon [Source: docs/kanban-architecture.md#Phase 1: Foundation].

**Data Models**: 
- **kanban_stages table schema** [Source: docs/kanban-architecture.md#Database Schema Design]:
  - `id` (BIGSERIAL PRIMARY KEY) - Stage identifier
  - `account_id` (BIGINT NOT NULL) - Account association with CASCADE delete
  - `name` (VARCHAR(255) NOT NULL) - Stage display name, unique per account
  - `color` (VARCHAR(7) NOT NULL DEFAULT '#6366f1') - Hex color code for UI
  - `position` (INTEGER NOT NULL) - Stage ordering, unique per account
  - `created_at`, `updated_at` (TIMESTAMP NOT NULL) - Rails timestamps

- **conversations table extension** [Source: docs/kanban-architecture.md#Modified Table: conversations]:
  - `kanban_stage_id` (BIGINT NULL) - Optional foreign key to kanban_stages
  - Foreign key constraint with SET NULL on stage deletion

**Database Constraints & Indexes** [Source: docs/kanban-implementation/01-database-migration.md]:
- **Unique Constraints**: (account_id, name), (account_id, position)
- **Check Constraints**: color ~ '^#[0-9A-Fa-f]{6}$' for valid hex colors
- **Performance Indexes**: 
  - `idx_kanban_stages_account_position` ON (account_id, position)
  - `idx_conversations_account_kanban_stage` ON (account_id, kanban_stage_id)
  - `idx_conversations_kanban_stage_updated` ON (kanban_stage_id, updated_at DESC)

**Model Validations & Behaviors** [Source: docs/kanban-architecture.md#KanbanStage Model]:
- **KanbanStage**: name presence/length/uniqueness, color hex format, position range 1-20
- **Position Management**: move_to_position! method handles position shifting logic
- **Real-time Broadcasting**: after_commit callbacks for ActionCable updates
- **Stage Deletion**: dependent: :nullify for conversations association

**Feature Flag Implementation** [Source: docs/kanban-architecture.md#Feature Flag Implementation]:
- **features_enabled JSONB column** on accounts table for flexible feature control
- **PostgreSQL function** account_has_feature(account_id, feature_name) for query optimization
- **Rollout percentage logic** for gradual deployment based on account_id modulo
- **Feature flag checks**: 'kanban_enabled', 'kanban_real_time_updates', 'kanban_advanced_filters'

**Migration Strategy** [Source: docs/kanban-implementation/01-database-migration.md]:
- **Primary Migration**: 20250115_create_kanban_stages.rb with table creation and conversation extension
- **Feature Flag Migration**: 20250115_add_kanban_feature_flag.rb with JSONB support
- **Default Stage Creation**: Automatic creation for all existing accounts during migration
- **Rollback Safety**: Complete rollback procedure with proper constraint removal order

**Performance Considerations** [Source: docs/kanban-architecture.md#Performance Architecture]:
- **Query Optimization**: KanbanQueryable concern with kanban_board_data method
- **Efficient Filtering**: apply_kanban_filters with proper index utilization
- **Conversation Grouping**: Optimized group_by operations for board loading
- **Index Strategy**: Composite indexes for common query patterns

**File Locations**:
- Migrations: `db/migrate/20250115_create_kanban_stages.rb`, `db/migrate/20250115_add_kanban_feature_flag.rb`
- Models: `app/models/kanban_stage.rb` (new), `app/models/conversation.rb` (extended)
- Concerns: `app/models/concerns/kanban_broadcastable.rb`, `app/models/concerns/kanban_queryable.rb`
- Feature Flags: `config/initializers/feature_flags.rb`
- Tests: `spec/models/kanban_stage_spec.rb`, `spec/models/conversation_spec.rb` (kanban extensions)

**Real-time Architecture** [Source: docs/kanban-architecture.md#Real-time Architecture]:
- **ActionCable Integration**: KanbanBroadcastable concern for consistent broadcasting
- **Event Types**: 'stage_created', 'stage_updated', 'stage_destroyed', 'conversation_stage_changed'
- **Channel Strategy**: Account-scoped channels "account_#{account_id}_kanban"
- **Broadcast Data**: Standardized event format with type, model, id, and serialized data

**Technical Constraints** [Source: CLAUDE.md]:
- **Database**: PostgreSQL with Rails 7.1 migration patterns
- **Backward Compatibility**: All changes must be additive, no breaking changes
- **Code Standards**: Follow RuboCop rules (150 character max line length)
- **Performance**: Migrations must use CONCURRENTLY for index creation in production
- **Testing**: Use RSpec following existing project patterns

**Integration Points**: 
- **Account Model**: has_many :kanban_stages association for account-based scoping
- **Activity Logging**: Stage changes create activity records for audit trail
- **Policy Framework**: Prepared for Pundit authorization policies in future stories
- **API Foundation**: Database layer ready for RESTful API controller implementation
